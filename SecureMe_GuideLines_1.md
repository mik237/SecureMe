SecureMe Vault System — Complete Concept & Architecture (Production Design)

Application Name: SecureMe
Application ID: me.secure.vault.secureme

SecureMe is a production-grade Android application designed to function as a zero-knowledge encrypted personal vault combined with an end-to-end encrypted file sharing platform. The application is developed using Kotlin as the programming language and Jetpack Compose for modern UI construction, while following Clean Architecture principles to ensure maintainability and scalability. The system adopts the MVI (Model-View-Intent) design pattern to manage state in a predictable and reactive manner, and Hilt is used for dependency injection across the entire application. Firebase serves as a backend infrastructure layer used only for authentication, encrypted key storage, metadata synchronization, and file transfer coordination. However, all cryptographic operations such as encryption, decryption, signing, and key management occur entirely on the client side. This ensures that no plaintext data or private cryptographic material is ever accessible to the server, making SecureMe a true zero-knowledge system.

The application architecture is divided into three main layers: Presentation, Domain, and Data. The Presentation layer contains Jetpack Compose UI screens, ViewModels, and UI state models that follow the MVI design pattern. Each screen exposes immutable UiState objects, processes user actions through UiIntent, and emits one-time events using UiEffect. This approach ensures predictable state flow and makes the application highly testable. The Domain layer contains business logic implemented as use cases, repository interfaces, vault orchestration logic, cryptographic workflows, and sharing processes. The Data layer implements repositories and manages all interactions with Firebase services, local storage, cryptographic modules, and metadata management. Cryptographic logic resides inside a dedicated crypto package, while Firebase operations are isolated within a firebase package, ensuring a strict separation between security logic and networking components.

When a user launches SecureMe for the first time, the application initiates an onboarding process that guides the user through creating a strong password. This password must satisfy strict security requirements such as minimum length, uppercase characters, lowercase characters, numbers, and symbols. The password itself is never stored directly anywhere in the system. Instead, the application derives a secure cryptographic key from the password using a strong key derivation function such as Argon2, with PBKDF2 available as a fallback if required. This derived key is then used to encrypt and protect the vault's master key. The master key acts as the root encryption key of the entire vault system.

During the onboarding process, SecureMe generates a master key which is used to encrypt vault metadata and all file encryption keys. The master key itself is encrypted using the password-derived key and then stored securely in Firebase as an encrypted object associated with the authenticated user. This design allows the master key to be recovered on a new device while still remaining protected by the user's password. Even Firebase cannot decrypt the master key because it only stores the encrypted version. The application also generates two asymmetric key pairs during onboarding: an X25519 key pair used for encryption and secure key exchange between users, and an Ed25519 key pair used for digital signatures and identity verification. The private keys are encrypted using the master key and stored securely in Firebase, while the public keys are stored in Firestore so other users can securely send encrypted data.

SecureMe operates around a single primary vault stored locally on the user's device. Instead of using a database container or encrypted archive file, the vault consists of individually encrypted files stored within a dedicated folder in external media storage. All local storage paths that are user-specific are scoped by the authenticated user's Firebase UID, ensuring complete isolation between multiple user accounts on the same device. The vault folder for each user is located at getExternalFilesDir(null)/SecureMe_Vault/{userId}/, and the vault metadata file is stored separately in internal app storage at filesDir/{userId}/vault_metadata.enc. This scoping ensures that if multiple accounts are used on the same device, no user's data can overwrite or interfere with another's. A .nomedia file is placed inside each user's vault folder to prevent the Android system and gallery applications from indexing or displaying encrypted files.

The vault structure contains encrypted files and a metadata file that maintains the index of all vault content. Each file imported into SecureMe is encrypted individually using AES-256-GCM encryption with a unique initialization vector and a randomly generated file encryption key. The encryption process is performed using streaming encryption to ensure large files such as videos can be processed efficiently without consuming large amounts of memory. Instead of loading entire files into memory, SecureMe reads the original file in small chunks, encrypts each chunk, and writes the encrypted output to the vault storage. This approach allows the application to securely handle files up to the defined limit of 200 MB without performance issues.

Every encrypted file stored in the vault has its own unique file encryption key. These file keys are not stored separately on the device in plaintext form. Instead, they are encrypted using the vault master key and saved within the encrypted vault metadata file. The metadata file acts as the vault index and contains essential information such as file identifiers, encrypted file keys, file types, storage paths, file sizes, creation timestamps, and ownership information. This metadata file is encrypted using the master key and stored in internal app storage at filesDir/{userId}/vault_metadata.enc, scoped per user, to ensure that even if someone accesses the storage folder directly, they cannot determine what files exist in the vault or how they are organized. Keeping metadata in internal storage rather than the shared vault folder provides an additional layer of isolation from other applications.

To prevent accidental data loss and improve reliability across devices, SecureMe synchronizes encrypted metadata to Firebase. Whenever the vault metadata changes, such as when files are added, deleted, or received from other users, the application encrypts the updated metadata and uploads it to Firebase as a backup. This ensures that if a user loses their device, uninstalls the application, or accidentally deletes local metadata, the vault structure can be restored from the encrypted backup stored in the cloud.

SecureMe uses Firebase Authentication as the identity system for users. Each user signs in using Firebase authentication methods such as email authentication. Once authenticated, the application retrieves the encrypted master key and encrypted private keys associated with that account. The user must then enter their vault password, which is used to derive the decryption key required to unlock the master key. Once the master key is decrypted, the application can restore the vault metadata, private keys, and file encryption keys. This process allows the user to access their encrypted vault on any device while maintaining strict security guarantees.

One of the core features of SecureMe is encrypted file sharing between users. The application allows users to send encrypted files directly to other SecureMe users without relying on external messaging platforms. When a user shares a file, the encrypted file stored locally is uploaded to Firebase Storage without decrypting it. The file encryption key associated with that file is encrypted using the recipient's public encryption key derived from their X25519 key pair. This ensures that only the intended recipient can decrypt the file key. Additionally, the sender signs the sharing transaction using their Ed25519 signing key, which allows the recipient to verify the authenticity of the file and confirm that it originated from the expected sender.

When the receiving user downloads the shared file, their SecureMe application retrieves the encrypted file and encrypted file key from Firebase. The application uses the recipient's private encryption key to decrypt the file key and then re-encrypts that file key using the vault master key before storing it in the metadata file. The encrypted file itself is then moved into the recipient's UID-scoped local vault folder and becomes part of that user's vault, allowing the file to be managed, previewed, or shared further if desired.

SecureMe also provides a secure file import workflow designed to eliminate duplication and improve user privacy. When importing files from the device gallery or file manager, the application offers the user the option to move the file into SecureMe instead of copying it. In this workflow, the original file is encrypted and stored in the vault, and then the original unencrypted file is removed from the device storage if the user confirms the action. This ensures that sensitive files do not remain exposed in the device gallery after being secured inside the vault.

The application includes a Home screen that organizes vault content into multiple tabs such as Images, Videos, Documents, and Other Files. These tabs are generated dynamically based on file types stored in the vault metadata rather than relying on folder structures that could be modified externally. This ensures that even if someone attempts to tamper with storage folders, the application can still accurately determine which files belong to each category using metadata.

SecureMe implements a secure identity verification system for contacts. Each user has a unique identity derived from their public keys, and the application generates a fingerprint by hashing the public keys using SHA-256 and displaying a shortened version of the hash in a human-readable grouped format. This fingerprint can be compared between users through an out-of-band channel such as a phone call to confirm identity. Once verified, the contact is marked as trusted locally. If a contact's public key ever changes on the server, SecureMe detects the fingerprint mismatch and warns the user of a potential security risk, blocking encrypted file sharing until the identity is re-verified.

Security hardening is an important aspect of the SecureMe design. The application prevents screenshots on sensitive screens, avoids storing decrypted data on disk, clears temporary memory buffers after use, and ensures all cryptographic operations use secure random number generators. Clipboard operations containing sensitive data are cleared automatically, and debugging logs containing sensitive data are disabled in release builds. The vault automatically locks after a period of inactivity, requiring the user to re-enter their password to unlock the master key.

Because encrypted files are stored locally while keys and metadata are securely synchronized to Firebase, SecureMe achieves a balance between offline access and cloud recovery. Even if a user changes devices or reinstalls the application, they can log in using Firebase authentication, retrieve their encrypted keys and metadata backup, enter their vault password, and restore access to all encrypted files stored locally or downloaded again through sharing history.

Multi-user device safety is a first-class concern in SecureMe. Every local storage path that is user-specific is namespaced by the authenticated Firebase UID. This means that if multiple user accounts are used on the same physical device, each user's vault folder, metadata file, and temporary cache are completely isolated from one another. On logout, the SessionManager clears all in-memory keys but leaves the user's encrypted files and scoped metadata on disk untouched, as they are encrypted and safe at rest. This allows the user to return to their fully intact vault on the next login without any data loss. A different user logging in on the same device will only ever see and operate within their own UID-scoped directories and will never have access to, or accidentally overwrite, another user's data.

Overall, SecureMe is designed as a professional-grade encrypted vault and secure file transfer platform that combines modern Android development practices with strong cryptographic architecture. It leverages client-side encryption, file-level key management, secure metadata synchronization, public key identity verification, zero-knowledge cloud storage, and strict per-user local storage scoping to provide a robust privacy-focused system. The architecture ensures that even in the event of server compromise, device loss, unauthorized storage access, or multi-user device scenarios, encrypted data remains protected and accessible only to the rightful user through proper authentication and cryptographic verification.
